# 前端页面优化审查报告

更新时间：2026-05-24

## 1. 审查范围

本次审查覆盖当前项目的两个前端入口：

```text
frontend/      # 管理端 Vue Web，同时仍保留一套旧读者 Web 页面
miniprogram/   # 当前主维护的用户端 uni-app 工程
```

本次审查以代码结构、页面布局、样式一致性、交互完整度、移动端真机适配和后续可维护性为主。

本次没有新增业务功能，也没有调整后端接口。

## 2. 总体结论

当前前端已经达到“主流程可用”的状态。

管理端的图书、借阅、用户、分类、库位、ISBN 资料库、数据看板等页面功能比较完整，已经具备统一主题变量、通用链接样式、条码浮层、时间格式化等基础规范。

用户端已经切到 uni-app，并且图书大厅、图书详情、我的借阅、个人资料、扫码借阅、扫码归还等读者核心流程基本齐全。

下一阶段不建议继续堆新页面，而应该先做一次前端体验收口：

| 优先级 | 方向 | 目标 |
|---|---|---|
| P0 | 统一样式与状态 | 消除重复样式、内联样式、空白状态不一致 |
| P0 | 移动端真机体验 | 处理 tabBar/FAB/安全区/扫码确认流程 |
| P1 | 管理端表格体验 | 处理固定列、横向滚动、操作列、响应式对话框 |
| P1 | 看板交互可感知 | loading/empty/error、点击提示、图表异常兜底 |
| P2 | 代码维护性 | 抽组件、抽组合逻辑、减少 Sass 弃用警告 |

### 3.2 管理端主要问题

#### 问题 1：页面标题、工具栏、表格卡片样式仍重复

虽然已经有 `.app-toolbar` 和 `.app-table-card`，但很多页面仍在 scoped style 中重复定义：

```text
.page-title
.page-title::after
.toolbar
.table-card
.card
```

涉及页面包括：

```text
frontend/src/views/admin/BookList.vue
frontend/src/views/admin/BorrowList.vue
frontend/src/views/admin/UserList.vue
frontend/src/views/admin/CategoryList.vue
frontend/src/views/admin/ShelfList.vue
frontend/src/views/admin/IsbnMetadataList.vue
frontend/src/views/admin/Settings.vue
frontend/src/views/admin/Dashboard.vue
frontend/src/views/admin/UserDetail.vue
```

建议：

| 优先级 | 优化动作 |
|---|---|
| P0 | 新增全局 `.app-page-title`、`.app-page-title::after` |
| P0 | 页面统一使用 `.app-toolbar`、`.app-table-card`，删除重复 scoped 定义 |
| P1 | 抽一个 `AdminPageShell` 或 `PageSection` 组件，统一标题、筛选区、内容卡片间距 |

#### 问题 2：表格固定右列仍有潜在遮挡风险

之前借阅管理的固定列问题已经修过，但其他页面仍有 `fixed="right"`：

```text
BookList.vue
BookDetail.vue
CategoryList.vue
ShelfList.vue
UserList.vue
IsbnMetadataList.vue
```

固定列在暗色主题、横向滚动、表格宽度不足时，容易再次出现操作列遮挡、背景断层、滚动条压住内容等问题。

建议：

| 优先级 | 优化动作 |
|---|---|
| P0 | 只在确实需要的表格保留 `fixed="right"` |
| P0 | 操作列统一最小宽度，不要让内容贴边 |
| P1 | 对窄屏或小窗口时取消固定列，改为横向滚动 |
| P1 | 为所有表格统一 `empty-text`、行 hover、当前行、操作列样式 |

#### 问题 3：看板缺少 loading、empty、error 的完整状态

当前数据看板多个接口并行加载：

```text
stats
monthly-borrows
hot-books
book-stocks
inventory-health
shelf-capacity
exceptions
activity-feed
```

如果某个接口失败，图表区域可能表现为空白或停留旧数据，用户不知道是没数据还是请求失败。

建议：

| 优先级 | 优化动作 |
|---|---|
| P0 | 为每个图表卡片增加 loading 状态 |
| P0 | 图表数据为空时显示统一空状态 |
| P1 | 每个看板模块单独 try/catch，避免一个接口失败拖垮整个看板 |
| P1 | 图表标题旁增加“点击查看详情”提示或 hover hint |
| P2 | 侧边栏收缩、标签页切换后主动触发 ECharts resize |

#### 问题 4：主题覆盖存在较多 `!important`

全局样式里针对 ocean、graphite 等主题做了大量覆盖：

```text
html[data-theme='ocean'] ...
body[data-theme='graphite'] ...
```

其中部分规则使用 `!important`。这能快速压住样式，但后续容易和 Element Plus、scoped style、表格固定列样式互相抢优先级。

建议：

| 优先级 | 优化动作 |
|---|---|
| P1 | 把主题差异尽量下沉到 CSS 变量，不直接覆盖具体组件 |
| P1 | 减少 `!important`，避免后续修一个页面影响多个页面 |
| P2 | 建立“主题 QA 清单”：浅色、暗色、ocean、graphite 各检查关键页面 |

#### 问题 5：管理端对话框响应式不足

多处对话框使用固定宽度：

```text
width="420px"
width="520px"
width="540px"
width="600px"
width="620px"
```

管理端主要面向电脑，这不是严重问题，但在窄屏笔记本、浏览器缩放、分屏窗口下会不够稳。

建议：

| 优先级 | 优化动作 |
|---|---|
| P1 | 统一对话框 class，例如 `.app-dialog-md`、`.app-dialog-lg` |
| P1 | 设置 `max-width: calc(100vw - 32px)` |
| P2 | 表单字段较多的弹窗改为分组布局，减少拥挤 |

#### 问题 6：管理端仍保留旧读者 Web 页面

当前 `frontend/src/views/user/` 仍存在：

```text
BookHall.vue
MyBorrows.vue
UserProfile.vue
Layout.vue
```

但项目方向已经明确：用户端主维护 `miniprogram/` 的 uni-app。继续保留 Web 读者端会造成维护边界不清。

建议：

| 优先级 | 优化动作 |
|---|---|
| P1 | 明确 `frontend/src/views/user` 是保留预览、废弃、还是继续维护 |
| P1 | 如果废弃，隐藏 `/user/**` 路由或移动到 legacy 目录 |
| P2 | 如果继续维护，应同步样式规范与用户端业务逻辑，否则长期会分叉 |

## 5. 推荐优化顺序

### 第一阶段：低风险体验收口

目标：不改业务逻辑，只统一外观和页面状态。

建议任务：

```text
1. 管理端抽全局 page-title / toolbar / table-card 样式
2. 用户端移除模板内联样式，补齐统一 retry/loading/empty 组件
3. 绑定账号接口明确前继续不暴露入口
4. 检查所有 tabBar 图标和 FAB 在真机上的位置
5. 后续类型声明继续集中维护 `miniprogram/shims-uni.d.ts`
```

验收标准：

```text
所有页面标题样式一致
所有列表页 loading/error/empty 样式一致
用户端没有明显“开发中”入口暴露给普通用户
真机底部导航和扫码 FAB 不遮挡列表内容
构建不新增错误
```

### 第二阶段：管理端表格和看板优化

目标：让管理端在常用屏幕宽度下更稳。

建议任务：

```text
1. 梳理所有 fixed="right" 操作列
2. 对窄窗口取消固定列或优化横向滚动
3. 看板每个图表卡片加 loading/empty/error
4. 图表和可点击区域增加 hover hint
5. 对话框统一 max-width
```

验收标准：

```text
借阅管理、图书管理、用户管理、ISBN 资料库、库位管理没有操作列遮挡
看板接口失败时不会出现纯空白卡片
图表点击行为用户可感知
窄屏或浏览器分屏时对话框不溢出屏幕
```

### 第三阶段：用户端业务体验精修

目标：让读者端更像正式可用的移动产品。

建议任务：

```text
1. 抽 scanAndBorrow / scanAndReturn 公共流程
2. 实现个人资料编辑
3. 梳理绑定账号入口，未接微信登录前先隐藏
4. 图书封面加载失败时显示统一占位
5. 借阅卡片增加长按复制提示和可视化反馈
```

验收标准：

```text
扫码借阅和扫码归还文案一致
任意页面扫码失败都能手动输入
个人资料可编辑并保存
封面异常不破坏卡片布局
用户端 App 和微信小程序表现一致
```

## 7. 风险与注意事项

### 不建议马上做的事情

```text
不要马上大换皮
不要同时重构管理端和用户端所有页面
不要继续维护两套读者端而不说明边界
不要为了改 tabBar 继续来回移动 public/static 路径
```

### 应优先保持稳定的事情

```text
管理端主题变量体系
用户端 uni-app 单一源码方向
tabBar 使用 static/tab 路径
真机调试使用 lan 环境
后端接口保持稳定，Swagger 继续按环境变量控制
```

## 8. 建议下一步执行项

如果现在开始动手，建议按这个顺序做：

```text
1. 用户端：抽重试/骨架屏样式，继续减少模板内联样式
2. 用户端：真机复查 tabBar、FAB、详情页底部栏安全区
3. 管理端：统一 page-title/toolbar/table-card
4. 管理端：检查所有 fixed right 操作列
5. 管理端：给数据看板加 loading/empty/error
6. 最后统一跑 frontend build、mp-weixin build、app build
```

推荐第一轮只做第 1、2、3 项，风险最低，收益最大。
